MySQL:

索引：

索引的定义就是帮助存储引擎快速获取数据的一种数据结构，形象的说就是索引是数据的目录。

索引的分类：B+tree是选取最多的索引类型
按「数据结构」分类：B+tree索引、Hash索引、Full-text索引。
按「物理存储」分类：聚簇索引（主键索引）、二级索引（辅助索引）。
按「字段特性」分类：主键索引、唯一索引、普通索引、前缀索引。
按「字段个数」分类：单列索引、联合索引。

B+树索引

对于树的索引来说，树的高度越高，那么通常代表着所需要的IO次数也会增加

B+树分为主键索引和二级索引：
主键索引：
主键索引中叶子节点存放数据，非叶子节点只存放索引，而且每个叶子节点里的数据是按主键顺序存放的。每一层父节点的索引值都会出现在下层子节点的索引值中，因此在叶子节点中，包括了所有的索引值信息，并且每一个叶子节点都有两个指针，分别指向下一个叶子节点(也就是同一层的其他叶子节点)和上一个叶子节点(同上)，形成一个双向链表。

二级索引：
二级索引的B+树的叶子节点存放的是主键值，而不是实际数据。在查询过程中会先获取主键值，获取主键之后，再通过主键索引中的 B+Tree 树查询到对应的叶子节点，然后获取整行数据。这个过程叫「回表」，也就是说要查两个B+树才能查到数据
B+树读取一个结点可以当作是一次IO操作，而B+树存储千万级数据的时候只需要3-4层就可以满足了，因此效率会比较高

为什么MySQL选择B+树作为索引

首先，因为mysql中的数据是存储在磁盘中的，而磁盘的访问是很慢的，因此MySQL要考虑到尽可能减少磁盘的IO操作，所以使用B+树来作为索引，B+树作为一种多叉树作为索引有以下好处：
1：B+ 树的非叶子节点不存放实际的记录数据，仅存放索引，因此数据量相同的情况下，相比存储即存索引又存记录的 B 树，B+树的非叶子节点可以存放更多的索引，因此 B+ 树可以比 B 树更「矮胖」，查询底层节点的磁盘 I/O次数会更少。
2：B+ 树有大量的冗余节点（所有非叶子节点都是冗余索引），这些冗余索引让 B+ 树在插入、删除的效率都更高，比如删除根节点的时候，不会像 B 树那样会发生复杂的树的变化；
3：B+ 树叶子节点之间用链表连接了起来(每个叶子结点有两个指针，分别指向下一个和上一个叶子结点)，有利于范围查询，而 B 树要实现范围查询，只能通过树的遍历来完成范围查询，这会涉及多个节点的磁盘 I/O 操作，范围查询效率不如 B+ 树。

聚簇索引(主键索引)和非聚簇索引(二级索引)

聚簇索引：聚簇索引的 B+Tree 的叶子节点存放的是实际数据，所有完整的用户记录都存放在主键索引的 B+Tree 的叶子节点里；
二级索引：二级索引的 B+Tree 的叶子节点存放的是主键值，而不是实际数据。

索引失效有哪些

1：对索引使用左模糊匹配或者左右模糊匹配(右模糊匹配不会失效)
2：对索引进行表达式计算
3：对索引使用函数
4：对索引隐式类型转换

—————事务———————

事务有哪些特性

原子性：事务是最小执行单位，不允许分割。原子性确保操作要么全部完成，要么不起作用。

一致性：执行事务前后，数据保持一致，例如转账中，无论事务是否成功，两者的总金额不变。

隔离性：并发访问数据库时，一个用户的事务不被其它事务干扰，各并发事务之间数据库是独立的，每个事务都有一个独立的数据空间

持久性：事务一旦提交，它对数据库的改变是持久的，即使数据库发生故障也不应该对其有任何影响。

通过undo log(记载着修改前的信息)实现原子性、通过锁和MVCC机制来实现隔离性、通过redo log（每提交一个事务，该事务的日志就会写入到重做日志进行持久化）来实现持久性。

innoDB引擎是通过什么技术来保证事务

持久性是通过 redo log （重做日志）来保证的；
原子性是通过 undo log（回滚日志） 来保证的；
隔离性是通过 MVCC（多版本并发控制） 或锁机制来保证的；
一致性则是通过持久性+原子性+隔离性来保证；

数据库并行事务会引发什么问题

脏读：脏读就是事务A在还没有提交事务时，事务B访问了事务A修改过的数据，之后如果事务A发生了回滚，那么就表示事务B发生了脏读

不可重复读：指的是一个事务在多次读取同一个数据时，出现了前后两次读到的数据不一致的情况，这是因为事务B可能对数据进行了修改，并提交了事务，导致发生了不可重复读

幻读：表示在一个事务内多次查询某个符合查询条件的记录数量时，出现了前后两次的数量不一致的情况，就是第二个事务增加或者删除了一个记录，导致第一个事务发生了幻读

事务的隔离级别有哪些：

一共有四个隔离级别：读未提交、读提交、可重复读、串行化
读未提交（read uncommitted），指一个事务还没提交时，它做的变更就能被其他事务看到；
在这个隔离级别下可能会发生：脏读、不可重复读、幻读

读提交（read committed），指一个事务提交之后，它做的变更才能被其他事务看到；
在这个隔离级别下可能会发生：不可重复读、幻读

可重复读（repeatable read），指一个事务执行过程中看到的数据，一直跟这个事务启动时看到的数据是一致的，MySQL InnoDB 引擎的默认隔离级别；
在这个隔离级别下可能会发生：幻读，可重复读在一定程度上可以解决幻读问题(但不是完全解决，)

串行化（serializable ）；会对记录加上读写锁，在多个事务对这条记录进行读写操作时，如果发生了读写冲突的时候，后访问的事务必须等前一个事务执行完成，才能继续执行；
这个级别会解决所有因并行事务引发的问题

MySQL默认使用的可重复读隔离级别，但是他可以在很大程度上避免幻读问题(通过MVCC方式解决)

可重复读隔离级是由 MVCC（多版本并发控制）实现的，实现的方式是开始事务后（执行 begin 语句后），在执行第一个查询语句后，会创建一个 Read View，后续的查询语句利用这个 Read View，通过这个 Read View 就可以在 undo log 版本链找到事务开始时的数据，所以事务过程中每次查询的数据都是一样的，即使中途有其他事务插入了新纪录，是查询不出来这条数据的，所以就很好了避免幻读问题。

MVCC是什么

MVCC（Multi-Version Concurrency Control，多版本并发控制）是一种用于数据库管理系统的并发控制方法，他的主要目的是在高并发环境中实现数据库的高性能和数据的一致性，MVCC通过维护数据的多个版本，允许多个事务并发执行，从而避免了传统锁机制中的冲突和等待。

同一行数据平时发生读写请求时，会上锁阻塞住。但 MVCC 用更好的方式去处理读写请求，做到在发生读写请求冲突时不用加锁。

这个读是指的快照读，而不是当前读，当前读是一种加锁操作，是悲观锁。

MVCC机制是怎么具体实现的

实现原理主要是依赖记录中的 三个隐式字段，undo日志 ，Read View 来实现的。

InnoDB在每行数据都增加三个隐藏字段，隐含的自增ID（隐藏主键），如果数据表没有主键，InnoDB会自动以DB_ROW_ID生成一个聚簇索引，最近修改（修改、插入）事务ID：记录创建这条记录以及最后一次修改该记录的事务的ID，是一个指针，回滚指针，指向这条记录的上一个版本（上一个版本存储于，rollback segment里）

在MVCC机制下，创建一个新事务后，执行每个select语句前，都会创建一个读视图（Read View）用于确定哪些记录可以被当前记录看见，哪些不能看见。(这个read view是使用时间来确定那些能被看见那些不能被看见)

read view中主要包含四个主要内容： 表示创建ReadView时当前系统中活跃的事务的ID集合 (“活跃”指的就是，启动了但还没提交)，尚未提交事务中的最小事务id，表示创建ReadView时系统中应该分配给下一个事务的id值，当前最大事务ID+1，还有一个是创建这个read view的事务id

在创建Read View后，将记录中trx_id（最后修改当前事务的id）分为下面三种情况

  • 如果修改此记录的最后一个事务id小于当前ReadView中最小活跃的事务id，说明当前版本的记录是在创建此ReadVIiew前就已经提交了，所以此记录对当前事务是可见的。
  • 如果修改此记录的最后一个事务id大于等于将要分配的下一个事务id，说明此记录是在当前事务创建之后修改的，所以此记录不可见
  • 如果修改记录的最后一个事务id在下一个将要分配的事务id与最小活跃事务id之间，则查看修改此记录的最后一个事务id是否在活跃事务的集合里，如果在的话，说明修改当前事务的最后一个id仍然没有提交，此记录不可见，如果不在的话说明当前修改此记录的最后一个事务已经提交，则是可见的。

需要注意：在RC（读已提交）隔离级别下，事务对某条记录的第一次快照会创建一个ReadView,此后在调用快照读时，使用的还是同一个Read View。则在使用此Read View过程中，提交的事务，对于当前事务都是不可见的。如果此记录对于当前事务不可见，此时就会通过==回滚指针==向undo Log中寻找满足条件的历史版本。

在RR隔离级别下，快照读时才生成Read View，每次快照读时都会生成一个新的Read View，早于此Read View提交的事务都能看见。

MVCC的作用

  1. 提高并发性：MVCC允许多个事务并发执行而不需要对数据进行加锁，这大大提高了数据库的并发处理能力。每个事务看到的数据版本都是独立的，从而避免了事务之间的相互阻塞。
  2. 实现一致性读：MVCC支持一致性读，意味着一个事务在执行期间总是看到某个时间点的数据快照（即该事务开始时的数据状态），而不会受到其他事务提交的影响。这样保证了事务读取的数据是一致的。
  3. 减少锁的开销：传统的锁机制可能导致死锁和性能瓶颈，而MVCC通过多版本机制避免了这些问题。事务只需维护对各自版本的数据视图，而不需要频繁的锁操作。

MVCC的局限性

  • 空间开销：由于每次更新都会产生新版本，MVCC会增加存储的空间开销，需要定期清理不再需要的旧版本（例如通过垃圾回收）。
  • 复杂性：实现和维护多版本数据结构较为复杂，特别是在处理大量并发事务时。

MVCC的事务隔离级别是读提交和可重复读

MySQL的三大日志：

undo log 记录数据被修改前的样子，他的作用是防止丢失数据，用于事务失败后的回滚

undo log的记录时机：在mysql将要更新的数据加载到缓冲区Buffer pool 时，同时往 undo 日志文件中插入一条日志，也就是将 id=1 的这条记录的原来的值记录下来。

redo log 记录的是数据修改之后的值，不管事务是否提交都会记录下来

redo 日志文件是 InnoDB 特有的，他是存储引擎级别的，不是 MySQL 级别的，当服务器宕机mysql重启时，利用redo log日志内容恢复缓冲区数据，一般情况下，redo log Buffer 数据写入磁盘的策略是立即刷入磁盘

这里涉及到一个mysql服务器崩溃的情况：

1：如果在redo log持久化前，mysql服务器宕机了，mysql会认为本次事务失败，利用undo log日志恢复到更新前的样子

2：如果在redo log持久化后，mysql下次重启的时候会自动将redo log日志文件内容恢复到缓冲区中

binlog用于记录数据库执行的写入性操作(不包括查询)信息，以二进制的形式保存在磁盘中。

binlog是通过追加的方式进行写入的，可以通过max_binlog_size参数设置每个binlog文件的大小，当文件大小达到给定值之后，会生成新的文件来保存日志。

bin log一般用于主从复制和数据恢复

mysql从准备更新到事务提交全过程

1：首先执行器sql语句，来查询满足条件的数据，先是从缓存池中查询数据，如果没有就会去数据库中查询，如果查询到了就将其放到缓存池中
2：在数据被缓存到缓存池的同时，将原始数据写入 undo log 日志文件，
3：在 BufferPool 中更新数据，将更新后的数据添加到 redo log buffer 中，
4：完成以后就可以提交事务，在提交事务的前会做以下三件事
将redo log buffer中的数据持久化到 redo log 文件中
将本次操作记录持久化到 bin log日志文件中
将 bin log 记录结束的位置标记记录到redo log中，最后在 redo log 添加 commit 标记(表示事务完成)

  1. 通过索引找到要更改的记录，判断是否在Buffer Pool中，
  2. 执行器判断更新前的记录与更新后的记录是否一样，不一样让InnoDB执行相应更新操作
  3. 开启事务，记录undo log日志，并更新数据
  4. InnoDB根据WAL(write ahead logging)技术，先记录redo log日志，并将将redo log设置为prepare阶段，后面找合适时机修改真正的数据页
  5. 将相应的更新操作以二进制的形式写入到bin log中，提交事务。

Undo Log 链

我们知道Undo Log主要是用于数据库中事务回滚的，但在 MVCC 机制中也发挥着重要的作用，那什么是 Undo Log 链呢？

Undo Log 链是指在每个数据对象上维护的 Undo Log 记录链表。每张表都会有与之相对应的 Undo Log 链，用于记录修改前的数据信息（以方便数据进行回滚）

read view机制：ReadView其实就是一个保存事务ID的list列表。他是在事务开启的时候会生成一个readview记录的是本事务执行时，MySQL还有哪些事务在执行，且还没有提交。(当前系统中还有哪些活跃的读写事务)

数据库(MySQL)的内连接、外连接、全连接有什么区别

内连接：结合两张表的记录，返回两张表的交集部分

外连接：又分为左外连接和右外连接，左外连接表示左表的信息全部表示出来，右表只显示符合搜索条件的信息，不满足条件的地方为null，同理右外连接正好相反，将右表的信息全部表示出来，左表只显示符合搜索条件的信息，不满足条件的地方为null

全连接：就是将两张表合并，不管有的没的，都显示出来，也可以在后面加上order by来对数据进行排序

mysql和redis的区别

类型上：MySQL是关系型数据库,Redis是非关系型（缓存）数据库

需求上：MySQL和Redis因为需求的不同,一般都是配合使用

作用上：MySQL用于持久化的存储数据到硬盘,功能强大,但是速度较慢，Redis用于存储使用较为频繁的数据到缓存中,读取速度快

数据存放位置 ：MySQL数据放在磁盘，Redis数据放在内存

应用场景：MySQL和Redis都需要根据具体业务场景去选型

redis为什么快

Redis 运行比较快的原因主要有以下几点：

  1. 纯内存操作：Redis 将所有数据存储在内存中，这意味着对数据的读写操作直接在内存中进行，而内存的访问速度远远高于磁盘。这种设计使得 Redis 能够以接近硬件极限的速度处理数据读写。
  2. 单线程模型：Redis 使用单线程模型来处理客户端请求。这可能听起来似乎效率不高，但实际上，这种设计避免了多线程频繁切换和过度竞争所带来的性能开销。Redis 每个请求的执行时间都很短，因此在单线程下，也能够处理大量的并发请求。
  3. I/O多路复用：Redis 使用了 I/O 多路复用技术，可以在单个线程中同时监听多个客户端连接，只有当有网络事件发生时才会进行实际的 I/O 操作。这样有效地利用了 CPU 资源，减少了无谓的等待和上下文切换。
  4. 高效数据结构：Redis 提供了多种高效的数据结构，如哈希表、有序集合等。这些数据结构的实现都经过了优化，使得 Redis 在处理这些数据结构的操作时非常高效。

隔离性是解决什么问题？

解决的是在多用户环境下，多个事务同时访问和操作数据库，相互干扰，导致数据不一致的问题。

问题包括：

1、 脏读（读到的是脏数据，读到的是回滚前的数据。A对数据进行修改，B读到的是修改后的数据，但A将数据回滚，此时B读到的就是脏数据）。

2、 丢失数据（或者说覆盖数据。A先对数据进行修改，B再对数据进行修改，导致A对数据的修改被覆盖丢失）。

3、 不可重复读（类似幻读，不可重复读侧重于修改。A事务多次读同一个数据，B事务对这个数据的值进行了修改，导致A事务多次读到的数据值不同）。

4、 幻读（类似不可重复读，幻读侧重于插入。A事务多次对某张表进行读取，B事务向该表中插入数据，导致A事务在后续的读取中发现多了原本不存在的数据，就像出现幻觉一样）。

联合索引是什么？

定义：使用表中多个字段创建索引，就叫联合索引，也叫组合索引或复合索引。

使用场景：多条件查询，根据多个列进行查询。对多个列进行排序或分组操作。

最左匹配原则：

这个表示如果你的 SQL 语句中用到了联合索引中的最左边的索引，那么这条 SQL 语句就可以利用这个联合索引去进行匹配

MySQL怎么保证主从一致性

主服务器是写，从服务器是读，为了进行高并发，所以实现了主从

MySQL在主从复制（也称为主从同步）中，通过复制二进制日志（binlog）和重放这些日志来保证主从库的数据一致性。以下是MySQL如何保证主从库数据一致性的详细解释：

1. 主从复制的基本流程

  1. 主库记录二进制日志（binlog）：
    • 在主库上执行的所有数据修改操作（如INSERT、UPDATE、DELETE等）都会被记录在二进制日志中。二进制日志是按顺序记录的，保证了操作的执行顺序。
  2. 从库读取二进制日志：
    • 从库中的IO线程会从主库获取这些二进制日志，并将其写入本地的中继日志（relay log）中。
  3. 从库重放中继日志：
    • 从库中的SQL线程会读取中继日志，并按照日志记录的顺序依次执行这些SQL语句，从而使得从库的数据与主库保持一致。

2. 保证数据一致性的机制

MySQL通过以下机制来保证主从库的数据一致性：

  1. 事务保证：
    • MySQL使用事务的提交顺序来保证二进制日志中的顺序一致性。事务在主库上提交后，会被记录到二进制日志中，并按照提交的顺序进行复制。这意味着从库在重放这些事务时，也会按相同的顺序执行，确保数据一致。
  2. 二进制日志的顺序性：
    • MySQL的二进制日志是顺序写入的，保证了操作的顺序性。在从库上重放这些日志时，从库按照日志记录的顺序执行SQL操作，从而确保从库的数据状态与主库一致。

3. 一致性挑战

尽管MySQL有机制保证主从库的数据一致性，但在某些情况下可能会遇到挑战：

  1. 网络问题：
    • 网络延迟或丢包可能导致从库落后于主库，甚至导致数据不一致。半同步复制可以缓解这种问题，但仍然可能存在数据滞后。
  2. 从库崩溃或恢复：
    • 如果从库在处理二进制日志时崩溃，可能会导致数据丢失。恢复后从库可能需要重新同步，GTID可以帮助从库快速找到需要同步的事务。

MySQL日志模块中的redo log和bin log的二阶段提交

存在的问题：当执行一条更新语句时，redo log写入到磁盘后，此时MYSQL宕机，binlog还没来得及写入磁盘。MYSQL重启后，主库根据redo log恢复了数据（更新后 的），但是由于binlog没有这条日志，从库仍然是更新前的数据（旧值），就会导致主库的数据和从库的数据不一致。同理redo log没写入磁盘，而binlog写入到了磁盘，也会导致主从数据不一致问题。

解决办法：两阶段提交：

将 redo log 的写入拆成了两个步骤prepare和commit，这就是两阶段提交.

  • 第一个阶段：将事务对应的redolog日志标记为准备阶段（具体的刷盘时机取决于刷盘参数的设置）。如果此时MYSQL发生故障，重启后，发现没有对应bin log 此时会进行事务回滚。
  • 第二个阶段：此时会将事务的执行操作以二进制的形式写入到bin log日志文件中，并将redo log 设置为commit阶段，后面根据刷盘时机刷入到磁盘里。如果此时MYSQL发生故障，在重启MYSQL时会直接提交事务，恢复数据。

悲观锁和乐观锁：

个人理解一句话概括就是，在应用层面会造成线程阻塞的是悲观锁，而不会造成线程阻塞的是乐观锁，为什么这么说会在后续的内容中做详细介绍。

1.1、悲观锁
悲观锁是一种基于悲观态度的数据并发控制机制，用于防止数据冲突。它采取预防性的措施，在修改数据之前将其锁定，并在操作完成后释放锁定，以确保数据的一致性和完整性。悲观锁通常用于并发环境下的数据库系统，是数据库本身实现锁机制的一种方式。

在悲观锁的机制下，当一个使用者要修改某个数据时，首先会尝试获取该数据的锁。如果锁已经被其他使用者持有，则当前使用者会被阻塞，直到对应的锁被释放。这种悲观的态度认为数据冲突是不可避免的，因此在修改数据之前先锁定数据，以防止冲突的发生。

1.2、乐观锁
乐观锁是一种基于版本控制的并发控制机制。在乐观锁的思想中，认为数据访问冲突的概率很低，因此不加锁直接进行操作，但在更新数据时会进行版本比对，以确保数据的一致性。

乐观锁的原理主要基于版本号或时间戳来实现。在每次更新数据时，先获取当前数据的版本号或时间戳，然后在更新时比对版本号或时间戳是否一致，若一致则更新成功，否则表示数据已被其他线程修改，更新失败。

另外，CAS（Compare-And-Swap）是实现乐观锁的核心算法，它通过比较内存中的值是否和预期的值相等来判断是否存在冲突。如果存在，则返回失败；如果不存在，则执行更新操作。
